# 大学生创新创业训练计划项目 · 中期报告

---

## 一、项目中期总结

### 1. 项目过程

本项目自立项以来，严格按照任务书要求，采用"渐进式开发、分阶段推进"的策略，围绕"面向航空运维数据的后量子安全存储与可控共享"这一核心目标展开研究与实践工作。根据项目实际推进情况，团队对项目进度进行了合理调整，将项目整体分为两个阶段：第一阶段重点搭建基础加密平台，验证混合加密架构的可行性，并为L-ABE算法预留接口；第二阶段将实现L-ABE算法替换RSA-OAEP，完成从传统密码学到后量子密码学的核心升级。

在项目启动阶段，团队深入研读任务书，明确了"基于格的属性基加密（L-ABE）封装AES密钥"这一核心技术创新点。考虑到L-ABE算法的理论复杂性和工程实现难度，团队制定了"先验证架构、后升级算法"的渐进式开发策略。这一策略的核心思想是：首先使用成熟的RSA-OAEP算法作为密钥封装的过渡方案，同时设计通用的密钥封装接口，为后续L-ABE替换预留清晰的扩展点。通过完整实现AES-256-GCM数据加密与密钥封装的混合加密架构，验证系统整体架构的可行性；待架构稳定后，只需实现L-ABE密钥封装器并替换，即可实现后量子安全升级。

在技术选型阶段，团队经过充分调研，确定后端采用Spring Boot 4.x框架配合BouncyCastle密码库，前端采用原生JavaScript配合Web Crypto API实现浏览器端加密。Spring Boot作为企业级Java框架，生态完善、社区活跃，能够快速构建RESTful API；BouncyCastle是业界广泛使用的开源密码库，支持AES、RSA等主流算法，且持续跟进NIST后量子密码标准，为后续L-ABE实现提供了良好的技术基础。Web Crypto API是浏览器原生支持的加密接口，无需引入第三方库即可实现RSA、AES等算法，性能优异且安全性有保障。数据库选用H2内存数据库，轻量级且易于部署，适合原型验证阶段使用。

在系统设计阶段，团队设计了完整的混合加密架构。数据层采用AES-256-GCM对称加密算法，用于加密大量QAR数据文件；密钥层采用密钥封装机制（Key Encapsulation Mechanism），通过统一的KeyWrapper接口封装AES密钥，实现密钥的安全分发。当前阶段使用RSA-OAEP（2048位）作为密钥封装的具体实现，后续只需实现L-ABE版本的KeyWrapper接口即可完成升级。前后端通过握手协议协商会话密钥，后续所有API请求均使用AES-256-GCM加密传输，确保数据在传输过程中的机密性与完整性。

在开发实施阶段，团队按照"后端优先选型、前端分段跟进"的顺序推进。首先实现了Spring Boot后端的核心功能，包括用户认证与授权、文件上传下载、传输层加密、审计日志等模块。后端统一使用BouncyCastle作为加密Provider，确保加密实现的一致性和可维护性。传输层加密通过自定义Servlet Filter实现，对请求和响应进行透明的加解密处理，业务代码无需感知加密逻辑。团队特别设计了KeyWrapper密钥封装接口，当前实现为RsaKeyWrapper，后续可无缝替换为LabeKeyWrapper，为L-ABE升级提供了良好的架构基础。

前端开发阶段，团队实现了登录注册页面、工作台、管理员后台等核心界面。前端使用Web Crypto API实现RSA密钥对生成、AES加解密等加密操作，与后端握手协议对接，完成加密通信全流程。前端还实现了文件上传下载、数据统计、反馈提交等交互功能，用户体验流畅。

在测试与调试阶段，团队对系统进行了全面测试，包括功能测试、安全测试、性能测试等。测试过程中发现了多个问题，如Spring Security 6的XOR CSRF保护与SPA异步请求冲突、中文数据编码乱码、管理员审批接口403错误等，团队逐一分析原因并修复，确保系统稳定运行。

### 2. 进展情况，取得了哪些成绩，是否达到预期效果

截至目前，项目已按计划完成第一阶段的所有任务，取得了以下成绩：

**（1）混合加密架构完整实现并验证可行**

团队成功实现了AES-256-GCM数据加密与RSA-OAEP密钥封装的混合加密架构。数据加密采用AES-256-GCM算法，支持认证加密，能够同时保证数据的机密性与完整性，防止数据在传输过程中被窃听或篡改。密钥封装采用RSA-OAEP算法，服务端生成AES密钥后，用客户端公钥加密封装，客户端用私钥解密获得AES密钥，实现密钥的安全分发。整个握手协议与数据传输流程已完整打通，前后端加密通信正常工作，Wireshark抓包验证显示传输数据已加密，明文不可见，达到预期效果。

**（2）统一使用BouncyCastle密码库**

团队统一了后端加密实现，所有加密操作均使用BouncyCastle作为Provider。AesGcmUtil工具类使用BouncyCastle实现AES-256-GCM加密解密，RsaKeyWrapper使用BouncyCastle实现RSA-OAEP密钥封装。统一使用BouncyCastle的优势在于：一是确保加密实现的一致性和可维护性；二是BouncyCastle持续跟进NIST后量子密码标准，为后续L-ABE实现提供了良好的技术基础；三是BouncyCastle提供了丰富的密码学原语，便于后续扩展。达到预期效果。

**（3）密钥封装接口设计完成，为L-ABE替换预留扩展点**

团队设计了通用的KeyWrapper密钥封装接口，定义了wrap和unwrap两个核心方法。当前实现了RsaKeyWrapper，使用RSA-OAEP算法封装AES密钥。后续只需实现LabeKeyWrapper，即可无缝替换RSA-OAEP，完成L-ABE升级。这种接口抽象设计使得密钥封装算法的替换不影响业务层代码，只需修改依赖注入配置即可，架构设计合理，达到预期效果。

**（4）Web原型系统功能完备**

团队构建了完整的Web原型系统，包含用户端和管理员端两大模块。用户端提供登录注册、文件上传下载、数据统计、反馈提交等功能；管理员端提供人员管理、注册审批、文件管理、审计日志查询、数据导出等功能。系统采用前后端分离架构，后端提供RESTful API，前端通过AJAX异步调用，用户体验流畅。所有核心功能均已实现并测试通过，达到预期效果。

**（5）用户认证与授权机制完善**

团队实现了基于Session的无状态认证方案，用户提交工号和密码后，服务端验证密码哈希，创建Session令牌并存入HttpOnly Cookie，后续请求通过SessionAuthFilter自动验证。授权采用RBAC角色权限模型，区分ADMIN和USER两种角色，ADMIN角色拥有人员管理、注册审批、文件管理、数据导出等权限，USER角色仅拥有文件查看下载、个人信息修改等权限。权限控制精确到接口级别，未授权访问会被拦截，达到预期效果。

**（6）人员准入审批机制严格**

团队实现了"申请-审批"两阶段用户准入机制。用户注册时需填写工号、姓名、身份证后四位、联系方式、航司、职位、部门等信息，系统会与预设的人员档案库进行匹配验证，只有信息完全匹配才能创建申请记录。管理员在后台查看待审批申请，审核通过后才能正式创建用户账号。档案库数据从person_seed.csv文件动态加载，系统启动时自动初始化。这一机制确保了用户身份的真实性，达到预期效果。

**（7）审计日志系统完整可追溯**

团队实现了全局API审计日志系统，通过AuditLogFilter拦截所有API请求，记录时间戳、用户ID、请求方法、URI、响应状态等信息，存储于H2数据库。管理员可在后台查询审计日志，追溯所有用户操作，为安全事件调查提供依据。审计日志系统运行稳定，记录准确，达到预期效果。

**（8）传输层加密透明化实现**

团队通过自定义Servlet Filter实现了传输层加密的透明化处理。TransportCryptoFilter拦截所有API请求，检查请求头中的加密标识，若请求已加密则解密后再传递给业务层；业务层处理完成后，Filter再加密响应返回给客户端。这种设计使得业务代码无需感知加密逻辑，后续替换L-ABE算法时只需修改KeyWrapper实现，业务层无需改动，架构设计合理，达到预期效果。

**（9）中文编码问题全链路解决**

团队解决了中文数据在存储、传输、显示过程中的乱码问题。通过在pom.xml中配置UTF-8编码、在application.properties中配置数据库连接编码、在PersonSeeder中正确读取CSV文件编码、添加BOM处理等多重措施，确保中文数据在各环节正确显示。测试验证中文姓名、部门名称等信息均正常显示，达到预期效果。

**（10）Spring Security 6 CSRF兼容问题解决**

团队解决了Spring Security 6默认开启XOR CSRF令牌保护导致前端SPA异步请求被拦截的问题。通过配置CsrfTokenRequestAttributeHandler禁用XOR保护并解决延迟加载问题，确保管理员能够正常通过/驳回注册申请。同时统一将安全配置中的`.hasRole("ADMIN")`调整为`.hasAuthority("ROLE_ADMIN")`以精准匹配权限。CSRF防护与SPA兼容性均达到预期效果。

**（11）代码规模与质量达标**

团队完成了80+个Java源文件的开发，涵盖controller、service、model、repo、dto、config、security、transport、crypto、startup、util等多个包，代码结构清晰，职责分明。前端实现了5个核心页面，包含登录注册、工作台、管理员后台、数据管理、反馈等模块。代码遵循Spring Boot最佳实践，使用Spring Data JPA简化数据访问，使用Spring Security实现安全防护，代码质量良好，达到预期效果。

综上所述，项目第一阶段的所有任务均已完成，系统功能完备、运行稳定、安全性良好，达到预期效果。混合加密架构已验证可行，KeyWrapper接口设计为L-ABE升级预留了清晰的扩展点，为第二阶段核心升级奠定了坚实基础。

### 3. 项目进行中遇到的问题及可能的解决方案（包括调整）

**（1）Spring Security 6 CSRF保护与SPA冲突问题**

问题描述：在实现管理员审批功能时，前端SPA异步请求被Spring Security 6拦截，返回403 Forbidden错误。经分析发现，Spring Security 6默认开启XOR CSRF令牌保护，前端异步请求无法正确处理XOR编码的CSRF令牌。

解决方案：通过配置CsrfTokenRequestAttributeHandler禁用XOR保护，并解决延迟加载问题，确保前端能够正确获取和使用CSRF令牌。同时统一将安全配置中的`.hasRole("ADMIN")`调整为`.hasAuthority("ROLE_ADMIN")`以精准匹配权限。问题已解决，管理员审批功能正常工作。

**（2）中文数据编码乱码问题**

问题描述：在测试过程中发现，中文姓名、部门名称等信息在数据库存储、前端显示时出现乱码。经排查发现，问题出在多个环节：Maven资源过滤编码、数据库连接编码、CSV文件读取编码等。

解决方案：通过多重措施解决UTF-8编码问题：在pom.xml中配置`project.build.sourceEncoding`和`project.reporting.outputEncoding`为UTF-8；在application.properties中配置数据库连接URL添加`MODE=PostgreSQL`和`DATABASE_TO_UPPER=FALSE`；在PersonSeeder中使用UTF-8编码读取CSV文件，并添加BOM处理；配置全局字符编码过滤器。问题已解决，中文数据正常显示。

**（3）加密实现不统一问题**

问题描述：项目初期存在多种加密实现方式，部分使用Java标准Cipher，部分使用BouncyCastle，代码风格不一致，维护成本高，且不利于后续L-ABE集成。

解决方案：团队统一了加密实现，所有加密操作均使用BouncyCastle作为Provider。创建了AesGcmUtil工具类，统一封装AES-256-GCM加密解密操作；创建了KeyWrapper接口和RsaKeyWrapper实现，统一密钥封装逻辑。这种设计使得后续替换L-ABE时只需实现新的KeyWrapper，无需修改其他代码。问题已解决，加密实现统一且可维护。

**（4）密钥管理架构设计问题**

问题描述：项目初期AES密钥采用内存临时管理方式，Session中直接存储AES密钥对象，这种方式不利于后续L-ABE集成，且密钥生命周期管理不够灵活。

解决方案：重新设计了密钥管理架构，AES密钥不再在Session中存储，而是通过KeyWrapper接口进行封装传输。握手阶段服务端生成AES密钥后，立即通过KeyWrapper封装返回给客户端，服务端不存储明文密钥。后续每次请求时，客户端携带封装后的密钥（或通过其他机制），服务端解封后使用。这种设计使得密钥封装算法的替换变得简单，只需更换KeyWrapper实现即可。问题已解决，密钥管理架构合理。

**（5）L-ABE算法复杂度超出预期**

问题描述：在调研L-ABE算法实现时发现，PALISADE库主要基于C++，与Java后端集成需要JNI调用，复杂度较高；Lattigo库基于Go语言，需要通过微服务或gRPC与Java后端通信，增加了系统复杂度。

解决方案：调整技术路线，优先考虑使用BouncyCastle的后量子密码模块（BouncyCastle正在跟进NIST PQC标准），或者将L-ABE算法封装为独立微服务，通过REST API与主系统通信。团队已设计好KeyWrapper接口，无论采用哪种方案，都只需实现对应的KeyWrapper即可，对业务层无影响。团队将继续调研最优方案，在第二阶段确定最终技术路线。

**（6）数据存储安全问题**

问题描述：当前数据在服务端以明文存储，若服务器被入侵，数据可能泄露。这与项目"后量子安全存储"的目标存在差距。

解决方案：计划在第二阶段实现服务端数据静态加密，使用AES-256加密存储文件，密钥由属性权威管理，只有满足访问策略的用户才能解密。这将实现"传输加密+存储加密"的双重保护。

项目后期将重点推进L-ABE核心算法实现与系统集成，完成从传统密码学到后量子密码学的升级，具体工作计划如下：

**近期（1个月）：L-ABE核心算法研究与接口实现**

深入研究LWE/RLWE问题及其在ABE中的应用，在理解格密码学的数学基础上选型并搭建开发环境，优先评估BouncyCastle PQC模块的可用性，同时调研PALISADE和Lattigo的集成方案。实现LabeKeyWrapper类，继承KeyWrapper接口，实现L-ABE密钥封装逻辑。设计访问策略语言与属性映射规则，定义用户属性（航司、职位、部门等）与访问策略（如"QAR数据 AND 工程师"）的映射关系。

## 三、项目进行过程中遇到的问题及需要的支持

**（1）L-ABE算法实现难度较大**

L-ABE算法基于格密码学，数学理论复杂，工程实现难度大。PALISADE库主要基于C++，与Java后端集成需要JNI调用，开发效率低；Lattigo库基于Go语言，需要通过微服务或gRPC通信，增加了系统复杂度。团队需要更多时间学习和调试格密码库的使用。

**需要的支持**：希望指导老师能够提供L-ABE算法实现的参考资料或示例代码，推荐合适的Java格密码库，指导JNI或微服务集成的技术方案。

**（2）性能测试与优化经验不足**

团队在性能测试与优化方面经验不足，对于如何评估L-ABE算法的性能开销、如何优化加密解密流程、如何处理大文件上传等问题缺乏经验。

**需要的支持**：希望指导老师能够提供性能测试的方法论和工具推荐，指导性能优化的思路和技巧，帮助团队制定合理的性能测试方案。

**（3）安全性分析与渗透测试能力有限**

团队在安全性分析与渗透测试方面能力有限，对于如何验证系统的抗量子能力、如何进行全面的渗透测试、如何发现和修复安全漏洞等问题缺乏经验。

**需要的支持**：希望指导老师能够提供安全性分析的方法论和工具推荐，指导渗透测试的流程和技巧，帮助团队全面评估系统安全性。

**（4）项目时间紧张**

项目时间紧张，需要在有限时间内完成L-ABE算法实现、系统集成、性能测试、安全性分析等多项工作，任务繁重。
