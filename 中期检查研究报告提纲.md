## 二、项目背景与意义

### 2.1 研究背景

#### 2.1.1 航空运维数据的安全挑战
- QAR（快速访问记录器）数据是民航飞行品质监控的核心资产，涉及飞行高度、速度、姿态、发动机参数等敏感信息
- 航空运维数据通常需长期保存（10-30年），面临"先截获、后破解"的长期保密威胁
- 数据需在航空公司、空管局、维修单位等多部门间共享，缺乏细粒度管控手段

#### 2.1.2 量子计算对传统密码体系的冲击
- 现有公钥加密体系（RSA、ECC）面临被Shor算法破解的风险
- NIST已于2024年8月正式发布首批三项后量子加密标准（FIPS 203, 204, 205）
- NSA发布的CNSA 2.0套件要求2030年前开始淘汰传统算法，2035年前完成PQC全面迁移
- 我国依据《密码法》加速推进抗量子算法国标化进程

#### 2.1.3 属性基加密与格密码学的融合趋势
- 属性基加密（ABE）可实现"数据随策略流动"的细粒度访问控制，与航空数据跨部门共享需求高度契合
- 传统ABE基于配对等经典难题假设，无法抵抗量子攻击
- 格基（LWE/RLWE）ABE结合了格密码学的抗量子特性与ABE的灵活访问控制能力，成为研究热点

### 2.2 研究意义

#### 2.2.1 理论意义
- 探索基于格的属性基加密（L-ABE）在航空数据安全中的工程化应用范式
- 研究混合加密架构中L-ABE封装AES密钥的方案设计与安全性分析
- 为后量子密码学在行业数据安全领域的迁移提供技术验证

#### 2.2.2 实践意义
- 为航空公司QAR数据提供具备后量子安全潜力的管理平台
- 实现基于属性策略的细粒度可控共享，替代粗粒度RBAC
- 为行业应对PQC迁移路线图提供具有前瞻性的解决方案

## 三、研究目标与内容

### 3.1 总体目标

构建面向航空QAR数据的后量子安全存储与可控共享原型系统，采用**混合加密架构**：

```
数据层：AES-256-GCM 对称加密（高效加密大量数据）
密钥层：L-ABE 属性基加密（封装AES密钥，实现策略化访问控制）
```

### 3.2 分阶段目标

#### 第一阶段（中期前，已完成）：基础加密平台搭建
1. 实现 AES-256-GCM 数据加密流程
2. 采用 RSA-OAEP 作为密钥封装的过渡方案，验证混合加密架构可行性
3. 构建 Web 原型系统，实现数据上传加密、下载解密全流程
4. 实现用户认证、审批、审计等基础安全功能

#### 第二阶段（中期后，待完成）：L-ABE 核心升级
1. 基于 PALISADE/Lattigo 开源库实现 L-ABE 算法
2. 将 RSA-OAEP 密钥封装替换为 L-ABE 密钥封装
3. 设计支持复杂逻辑的属性访问策略（如"QAR数据 AND 工程师"）
4. 构建属性权威、数据所有者、数据使用者三方架构
5. 完成与传统方案的性能对比测试分析

### 3.3 研究内容

#### 3.3.1 混合加密架构设计
- AES-256-GCM 数据加密方案
- 密钥封装机制设计（RSA-OAEP → L-ABE 渐进式替换）
- 会话密钥管理与生命周期控制
- 前后端加密通信协议设计

#### 3.3.2 L-ABE 算法研究与实现
- LWE/RLWE 问题理论分析
- 基于 PALISADE/Lattigo 的 L-ABE 算法实现
- 访问策略表达与属性映射
- 密钥生成、加密、解密流程设计

#### 3.3.3 属性基访问控制系统
- 属性权威（Attribute Authority）架构设计
- 用户属性与航空业务角色的映射（航司、职位、部门等）
- 访问策略制定与策略语言设计
- 属性撤销与密钥滥用追责机制

#### 3.3.4 系统安全与工程化
- 用户认证与审批机制
- 全局审计日志系统
- CSRF/XSS 等Web安全防护
- 性能测试与安全性分析

### 4.2 关键技术实现

#### 4.2.1 混合加密协议

**握手阶段（当前：RSA-OAEP → 后续：L-ABE）：**

```
客户端                                         服务端
  │                                              │
  │  1. 生成RSA-2048密钥对                        │
  │     [后续: 提交用户属性凭证]                    │
  │                                              │
  │ ──── POST /api/transport/handshake ────────►  │
  │      {clientPublicKey, protocol:"tls"}        │
  │      [后续: {attributes, policy}]             │
  │                                              │  2. 生成AES-256密钥
  │                                              │  3. RSA-OAEP封装AES密钥
  │                                              │     [后续: L-ABE封装AES密钥]
  │ ◄─────────── 握手响应 ─────────────────────  │
  │      {sessionId, wrappedKey, expiresAt}       │
  │                                              │
  │  4. 用私钥解密获得AES密钥                      │
  │     [后续: 用属性密钥解密获得AES密钥]           │
  │  5. 建立加密会话                               │
```

**数据传输阶段（AES-256-GCM加密）：**

```
客户端                                         服务端
  │                                              │
  │  · 生成随机IV (12字节)                         │
  │  · 构造AAD = "METHOD /path"                   │
  │  · AES-256-GCM加密请求体                       │
  │  · 发送 {iv, ciphertext}                      │
  │ ────────────────────────────────────────────► │
  │                                              │  · AES-256-GCM解密
  │                                              │  · 处理业务逻辑
  │                                              │  · AES-256-GCM加密响应
  │ ◄──────────────────────────────────────────── │
  │  · AES-256-GCM解密响应体                       │
  │  · 获得明文数据                                │
```

**核心代码实现（服务端）：**

| 类 | 职责 | 后续改造点 |

| `TransportCryptoFilter` | 请求/响应加解密Servlet过滤器 | 无需修改 |
| `AesGcmUtil` | AES-256-GCM加密/解密 | 无需修改 |
| `RSAUtil` | RSA-OAEP加密/解密（过渡方案） | 替换为 L-ABE 实现 |

**核心代码实现（客户端）：**

| 模块 | 职责 | 后续改造点 |
|------|------|-----------|
| `transport.js` | Web Crypto API 密钥协商与AES加解密 | 握手阶段适配L-ABE |

#### 4.2.2 用户认证与授权

**认证流程：**
1. 用户提交工号+密码
2. 服务端验证密码哈希
3. 创建Session令牌，存入HttpOnly Cookie
4. 后续请求通过SessionAuthFilter自动验证

**当前授权模型（RBAC，后续升级为ABE）：**
- ADMIN角色：人员管理、注册审批、文件管理、数据导出
- USER角色：文件查看与下载、个人信息修改

**后续属性策略模型（ABE）：**
- 用户属性：航司、职位、部门、数据类型权限等
- 访问策略示例：`"QAR数据" AND "工程师"` 或 `"CAUC" AND ("机长" OR "副驾驶")`
- 数据所有者制定策略 → 属性权威签发属性密钥 → 用户属性满足策略方可解密

#### 4.2.3 人员准入审批机制

**注册流程：**
```
用户填写信息 → 档案库匹配验证 → 创建申请记录 → 管理员审批 → 创建用户账号
```

**档案库匹配验证：**
- 工号、姓名、身份证后四位、联系方式、航司、职位、部门
- 系统启动时从person_seed.csv加载档案数据
- 注册信息必须与档案库完全匹配

#### 4.2.4 审计日志系统

- `AuditLogFilter`：全局Servlet过滤器，拦截所有API请求
- 记录内容：时间戳、用户ID、请求方法、URI、响应状态
- 存储于H2数据库，支持管理员查询

## 六、安全性分析

### 6.1 当前安全能力（第一阶段）

| 安全维度 | 威胁 | 当前防护 | 后续增强 |
|---------|------|---------|---------|
| 数据机密性 | 传输窃听 | AES-256-GCM应用层加密 | — |
| 数据完整性 | 数据篡改 | GCM认证标签验证 | — |
| 密钥安全 | 密钥泄露 | RSA-OAEP封装，私钥不离开浏览器 | **L-ABE属性策略封装** |
| 访问控制 | 未授权访问 | RBAC角色权限 | **ABE属性策略** |
| 抗量子 | 量子计算攻击 | ❌ RSA/ECC不抗量子 | **✅ L-ABE基于格难题** |
| 认证安全 | CSRF/劫持 | HttpOnly Cookie + CSRF令牌 | — |
| 审计追溯 | 操作不可追溯 | 全局审计日志 | — |

### 6.2 L-ABE升级后的安全增强（第二阶段预期）

| 增强维度 | 说明 |
|---------|------|
| **抗量子安全** | L-ABE基于LWE/RLWE格难题，可抵抗量子计算攻击 |
| **细粒度访问控制** | 从RBAC的"角色级"升级为ABE的"属性级"策略控制 |
| **策略化密钥分发** | 数据所有者制定访问策略，属性权威签发密钥，无需点对点密钥协商 |
| **前向安全性** | 属性撤销后，已分发密钥无法解密新密文 |

## 七、存在的问题与后续计划

### 7.1 当前存在的问题

1. **密钥封装未实现后量子安全**：当前采用RSA-OAEP封装AES密钥，不具备抗量子能力，需替换为L-ABE
2. **访问控制粒度不足**：当前RBAC仅支持角色级控制，无法实现属性级的细粒度策略
3. **属性权威架构缺失**：尚未构建属性权威、数据所有者、数据使用者三方架构
4. **数据存储安全**：当前数据在服务端以明文存储，缺乏静态加密
5. **HTTPS部署**：当前为HTTP开发模式，生产环境需部署HTTPS
